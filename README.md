# 智能机房/温室环境监控与安防联动系统

本项目是嵌入式系统课程大作业，旨在利用 **ESP32** 开发板构建一个具备环境感知（温湿度）、安全防御（红外检测）、边缘决策以及 Web 远程控制能力的物联网智能节点。

---

## 1. 硬件采购清单 (Bill of Materials)

| 类别 | 零件名称 | 推荐规格 | 数量 | 用途 |
| :--- | :--- | :--- | :--- | :--- |
| **主控** | ESP32 开发板 | NodeMCU-32S (带 Wi-Fi/蓝牙) | 1 | 系统核心，运行 Web Server 与状态机 |
| **感知** | 温湿度传感器 | DHT11 | 1 | 采集机房/温室实时环境数据 |
| **感知** | 人体红外传感器 | HC-SR501 | 1 | 安防监测，触发硬件外部中断 |
| **执行** | 继电器模块 | 5V 1路 (带光耦隔离) | 1 | 强弱电隔离，控制散热风扇 |
| **执行** | 散热风扇 | 5V 小型直流风扇 | 1 | 物理干预，自动/手动散热 |
| **执行** | 有源蜂鸣器 | 3.3V/5V 通用 | 1 | 发现闯入时进行声学报警 |
| **配件** | 基础包 | 面包板、杜邦线、USB线 | 1套 | 电路连接与供电 |

## 2. 团队分工与协作模式 (Team Roles)

考虑到两人异地，我们采用 **“云端开发 + 本地集成”** 的模式：

* **硬件工程师 (在校)**：负责物理接线、硬件调试、最终代码烧录及实测。
* **后端工程师 (远程)**：负责 ESP32 核心 C++ 逻辑、FSM 状态机、RESTful API 接口开发。
* **前端工程师 (远程)**：负责 HTML/JS 监控面板开发（使用 Mock 数据进行 UI 调试）。

## 3. 详细开发路线图 (Roadmap)

### 阶段零：项目预启动与契约约定 (Pre-kickoff) 必看
> **目标**：在写下第一行代码前，统一三方标准，消除团队依赖，确保大家能同时开工。
#### 例如：
- [ ] **硬件引脚约定 (Pin Mapping)**：
  * DHT11 数据引脚 -> `GPIO 4`
  * HC-SR501 红外中断引脚 -> `GPIO 14`
  * 继电器控制引脚 -> `GPIO 5`
  * 蜂鸣器报警引脚 -> `GPIO 12`
- [ ] **前后端接口约定 (API Contract)**：
  * **GET `/api/status`** (前端拉取当前状态)：
    * 约定返回 JSON 格式：`{"temp": 26.5, "humi": 60.0, "mode": "AUTO", "fan_status": "OFF", "alarm": false}`
  * **POST `/api/control`** (前端下发控制指令)：
    * 约定发送数据：`cmd=auto` 或 `cmd=on` 或 `cmd=off`
#### 详情见docs目录

### 第一阶段：物理连接与环境搭建 (Hardware Layer)
- [ ] 按照引脚表完成传感器与 ESP32 的连接，确保接线稳固。
- [ ] 硬件同学拍摄接线图并上传至 `hardware_docs/`。

### 第二阶段：底层驱动开发 (BSP/HAL)
- [ ] 封装 DHT11 非阻塞读取函数。
- [ ] 配置 HC-SR501 外部中断（上升沿触发）。
- [ ] 编写继电器与蜂鸣器的基本控制逻辑。

### 第三阶段：核心状态机设计 (Task Layer)
- [ ] 实现 **有限状态机 (FSM)**：包含 `AUTO`（自动阈值）、`MANUAL_ON`（强制开启）、`MANUAL_OFF`（强制关闭）三种状态。
- [ ] 设定手动控制优先级高于自动逻辑。

### 第四阶段：网络通信与 API 构建 (App Layer)
- [ ] 配置 ESP32 连接局域网 Wi-Fi。
- [ ] 开发 `/api/status` (GET) 和 `/api/control` (POST) 接口。

### 第五阶段：前端 Web 交互 (B/S Architecture)
- [ ] 编写异步刷新页面（Fetch/AJAX），实现温湿度数据无感更新。
- [ ] 将前端代码挂载至 ESP32 的 Flash 文件系统中。

### 第六阶段：集成测试与优化
- [ ] 进行高频请求下的稳定性测试。
- [ ] 优化传感器消抖算法，防止风扇频繁启停。

## 4. Git 协作规范 (Git Guidelines)

### 4.1 提交信息格式 (Commit Message)
必须遵循以下格式：`类型: 描述`
* `feat`: 新增功能（如 `feat: 增加红外中断逻辑`）
* `fix`: 修复 Bug（如 `fix: 解决网页数据不刷新的问题`）
* `docs`: 文档更新（如 `docs: 更新硬件清单`）
* `refactor`: 代码重构（优化结构但不改功能）

### 4.2 开发流程
1.  **Pull**: 每天开始写代码前先执行 `git pull` 同步队友进度。
2.  **Branch**: 严禁在 `main` 分支直接改代码。请创建新分支：`人名/功能`（如 `liutaobo/feat-api`）。
3.  **PR**: 完成代码后，在网页端发起 **Pull Request**，由至少一名其他队友 Review 后合并。
